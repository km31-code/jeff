# Jeff Apex — The Jarvis Plan

This document is the product definition and execution plan for Jeff v3 ("Apex"):
the version of Jeff that is no longer an ambient assistant with good instincts,
but a genuine Jarvis-class coworker. It supersedes the phase numbering in
PHASES.md, PHASES_NEXT.md, and PHASES_TRANSFORMATION.md as the forward plan.
VISION.md and CHARACTER.md remain authoritative for what Jeff is and who Jeff is.
This document defines what Jeff becomes and exactly how we get there.

Revision 2. Changes from revision 1: Part V is rewritten as a fully
execution-ready roadmap — every milestone now carries goal, scope, files,
dependencies, done-criteria, and a check script, so a new engineer or agent can
pick up any milestone cold. Part VIII ("the initiative gap") is new: it
specifies the eight capabilities that separate an excellent ambient coworker
from genuinely Jarvis. Three of them are promoted into the Apex critical path
as milestones C7 (override channel), D8 (speculation scheduler), and D9
(self-extension); the remaining five form the post-Apex epochs F–H. Parts VI
and VII are extended accordingly.

The thesis has not changed and does not need to: a coworker, not a tool. Already
present, already aware, interruptible in both directions, able to carry real work
in parallel, speaking only when something is worth saying. That thesis is right.
What has to change is the altitude of every capability underneath it. The current
system is the correct skeleton wearing an underpowered nervous system. Apex
replaces the nervous system.

**How to use this document if you are new here:**
1. Read VISION.md and CHARACTER.md in full. They define what Jeff is and who
   Jeff is. Nothing in this plan overrides them.
2. Read Parts I–IV of this document for the diagnosis, the bar, the pillars,
   and the architecture.
3. Go to Part V. Find the first milestone whose status is "not started" and
   whose dependencies are complete. Read its spec, read the pillar it serves,
   propose your implementation plan, get approval, execute, verify, commit.
4. The execution rules at the top of Part V are binding.

## Part I — Honest audit: where Jeff actually is

Thirty-one phases are built and committed. That work is real and most of it
survives into Apex. Inventory of what genuinely exists today:

**Presence and shell (keep, polish)**
- Tauri app, tray presence, global hotkey, frameless overlay, focus-preserving
  summon, single instance, close-to-hide, launch at login, session restore
- Signed/notarized universal macOS build, auto-update, release CI

**Interaction (keep the pipeline shape, replace the internals)**
- Streaming LLM → phrase-chunked TTS with barge-in both directions
- Whisper STT with Web Speech partial-transcript early routing
- Model-based intent classifier with keyword fallback
- Typing-activity-aware speech delay, brevity filter, voice selection

**Awareness (keep the architecture, replace the heuristics)**
- Workspace watcher with auto-ingest and embeddings (local RAG)
- Active window title polling (AX), opt-in document text polling (AX)
- Selection capture (hotkey + browser extension bridge)
- SituationalSnapshot assembled from deterministic signals, snapshot confidence
- Synthesis loop: should_speak_proactively + synthesized one-message delivery,
  synthesis_log audit trail, conversation-shaped proactive bubbles
- Calendar next-event context (EventKit, read-only, transient)

**Character and relationship (keep, deepen)**
- character.rs owns all system prompts; filler stripping; assessment-first output
- Behavioral user model (style, delegation, rhythm, rubrics)
- Relational model (stated goals, struggle patterns, collaboration style EMA,
  trust metrics), opinion-strength adaptation

**Work execution (replace with a real agent runtime)**
- Multi-step subtask chains (max 5 steps, 3 step types), approval-gated file
  writes, audit logs, cross-task collision detection, workload summary
- Live app edit path via browser extension with anchor validation and receipts

**Trust surface (keep, extend)**
- Privacy Center with per-surface toggles, clear-task / clear-all, audit views
- Keychain key storage, onboarding wizard, local-first SQLite

Phases 32–34 (character eval harness, native document write-back, web retrieval)
are specified but not implemented. Their intent is absorbed into this plan; their
specs remain useful reference and are cited where absorbed.

### The eight ceilings

The honest assessment. Each of these caps the felt experience regardless of how
well the surrounding code is written:

1. **The brain is a budget model.** gpt-4o-mini does everything: chat, judgment,
   drafting, classification, reorientation. Jeff's opinions, the core of the
   character, are generated by a model chosen for latency and price, not
   intelligence. A coworker is only as compelling as its judgment. This is the
   single highest-leverage ceiling in the system.

2. **Perception is string matching.** current_goal comes from scanning messages
   for the literal prefix "i'm working on". Content change detection compares the
   first 80 characters. Draft state is a word count. Struggle detection counts
   drift-trigger rows. These heuristics were the right way to bootstrap the
   architecture; they cannot see that a paragraph contradicts the thesis, that the
   user rewrote the same sentence six times, or that the document's structure
   drifted from the outline.

3. **Voice is stitched, not native.** STT → text → LLM → text → TTS is a
   half-duplex pipeline imitating conversation. Real-time speech-to-speech models
   now exist that hold full-duplex conversations with sub-second latency, natural
   prosody, and mid-utterance interruption as a native property rather than a
   cancellation-token achievement.

4. **The hands barely exist.** Write-back is specified for three apps via AX
   value replacement — a brittle mechanism that replaces the entire text buffer.
   There is no general ability to act: no browser control, no app scripting, no
   file operations beyond single approved writes, no system actions.

5. **Memory is a filing cabinet, not a memory.** Key-value profile rows and goal
   tables. No episodic record of what actually happened across sessions, no
   consolidation, no forgetting, no "remember when you restructured chapter 2 and
   it solved the same problem you're hitting now."

6. **Parallel work is a macro, not a colleague.** Five-step chains with three
   step types and a plan fixed at kickoff. A real delegate plans, acts, checks its
   own output, revises the plan when a step fails, and comes back with a finished
   deliverable and a view on it.

7. **The world is closed.** No web, no email, no calendar actions, no documents
   outside the watched folder. The target user's work lives across Gmail, Google
   Docs, Slack, and the browser. Jeff can see almost none of it and touch none.

8. **Trust is static.** Every action is approval-gated forever. A real coworker
   earns autonomy: the first draft gets reviewed line by line, the fiftieth gets
   skimmed, the five-hundredth gets a "thanks." Jeff has trust *metrics* but they
   only soften sentence phrasing. Nothing graduates.

None of these are indictments of the build. They are the natural state of a
system built phase-by-phase under strict scope control. The scaffolding is
correct. Apex is the capability pour.

Beyond the eight ceilings there is a ninth gap of a different kind — not a
capability that is underpowered, but a dimension that is absent: **initiative
depth**. Work done before being asked, presence beyond the desk, capability that
grows itself, interruption that can break through when it truly matters. That
gap is the subject of Part VIII.

## Part II — The bar: what "Jarvis-class" concretely means

The five felt properties from VISION.md, restated as Apex-level bars. Every
pillar and milestone in this plan traces to one or more of these.

| Felt property | v1/v2 delivery | Apex bar |
|---|---|---|
| Already present | Tray + hotkey + login item | Wake word or hotkey, sub-second voice engagement, presence across every surface the user works in (native apps, browser, email) |
| Already knows your task | Title + word count + goal-prefix match | Semantic understanding of the actual document, the diff since yesterday, the outline vs. the draft, the inbox thread that changes the plan, recalled across weeks |
| Interruptible both directions | Barge-in via cancellation tokens | Native full-duplex conversation; Jeff can also interject with judgment-weighted timing, mid-task, at the right moment — and break through unconditionally when something is genuinely urgent |
| Parallel work | 5-step chain, result card | "Handle the citations and check my numbers against the source" → a real agent works for 20 minutes with web access, verifies itself, and places results into the document on approval — plus work Jeff started on its own initiative before being asked |
| Initiates from judgment | Threshold triggers + one synthesized message | An interruption-economics engine on a frontier model: daily briefing, mid-work observations that are actually smart, end-of-day debrief, silence when silence is right |

### A day with Apex Jeff (the target experience, written as fact)

8:40am. You open the laptop. Jeff has already read the calendar, the overnight
email, and the state of the thesis draft. The overlay shows one line: "Draft
review with Sarah at 2 — chapter 3 is still missing the counterargument section
you outlined Tuesday. Want me to draft it from your notes while you do email?"
You say "yes, and flag anything in the inbox I actually need." You never opened
anything.

9:15am. Jeff interjects once, by voice, quietly: "Two things in email matter:
Sarah moved the review to 1:30, and the journal wants the revised abstract by
Friday. Everything else waits. Counterargument draft is ready when you want it."

10:00am. You're writing. Jeff is silent for 40 minutes. Then: "You've rewritten
that transition four times. The problem isn't the sentence — paragraph two makes
the claim this transition is trying to introduce. Cut it and the section flows."
You look. Jeff is right. You say "fix it," and the edit appears in the document
as a tracked change.

11:05am. You ask, "what did the reviewer say about my methods section last
month?" The answer is instant — not because the lookup is fast, but because
Jeff saw you rewriting the methods section twenty minutes ago and had already
pulled the reviewer's comments and checked them against your current text.
"Three concerns. You've addressed two. The sample-size one is still open —
your current numbers are the same ones they flagged."

1:25pm. "Review in five minutes. Since yesterday you added 900 words to chapter
3 and restructured the methods section — want a one-paragraph summary of what
changed to open with?" Yes. It's on your clipboard before you stand up.

4:30pm. "Before you close up: the citation check found two sources cited in
chapter 2 that aren't in the bibliography, and one page number that doesn't
match the source. I fixed the bibliography entries — proposal is waiting. The
page number I couldn't verify; the source PDF isn't in the workspace. Drop it
in and I'll finish the job."

That is the product. Every scene above maps to a specific pillar below, and
every scene is achievable within the Apex critical path (Epochs A–E). The
post-Apex epochs in Part VIII extend the same day beyond the desk.

### Pillar 1 — The Brain: frontier intelligence, tiered and routed

**Current:** gpt-4o-mini for every LLM call. Single provider, single model,
per-call HTTP with no caching strategy.

**Target:** Jeff thinks with a frontier model where judgment matters, a fast
model where speed matters, and a local model where privacy or cost matters —
routed automatically, invisible to the user.

**Design:**

- **Model router** (`model_router.rs`), replacing direct provider calls. Every
  LLM call site declares a *capability tier*, not a model:
  - `Tier::Reflex` — intent classification, filler checks, cheap summaries.
    Local model first (see below), fast cloud model fallback.
  - `Tier::Conversation` — chat turns, reorientation, voice. Fast frontier
    model (claude-haiku / gpt-4o-class realtime) with streaming.
  - `Tier::Judgment` — synthesis decisions, work-quality observations,
    interruption decisions, assessments. Mid frontier (claude-sonnet class).
  - `Tier::Craft` — drafting, revision, agent planning, verification passes.
    Top frontier (claude-sonnet/opus class), extended thinking enabled where
    the provider supports it.
- **Provider adapters** behind the existing `providers.rs` seams: OpenAI,
  Anthropic, and a local runtime. Config maps tier → provider/model and is
  user-overridable in settings, with sane defaults. Anthropic is the default
  for Judgment and Craft tiers; the character spec was written for a model
  that can actually hold a voice.
- **Local model runtime**: llama.cpp or MLX embedded (Apple Silicon first),
  running a small instruct model (3B–8B class) for Reflex tier and a local
  embedding model. Removes per-keystroke-adjacent costs, keeps the most
  frequent calls fully on-device, and makes classification latency ~10ms.
  Local runtime is optional at install (downloads on demand); cloud fallback
  always exists.
- **Prompt caching**: system prompts (character block, snapshot, relational
  context) restructured into stable-prefix blocks so provider prompt caching
  applies. Target: >70% cached input tokens on conversation turns.
- **Cost governor** (`cost_governor.rs`): per-day token budgets per tier,
  visible in settings ("Jeff has used $0.42 today"), degrade-gracefully rules
  (Judgment tier falls back to Conversation tier at budget, never silent
  failure). All spend visible in Privacy Center.
- **Latency contract** (extends Phase 17 budgets): first token < 700ms on
  Conversation tier, Judgment calls complete < 4s, Reflex < 100ms local.

**Exit criteria:**
1. Blind A/B on 20 real revision requests: Apex assessments preferred over
   v2 assessments in at least 16/20 by the user.
2. Every call site in the codebase routes through the model router; grep
   for direct model-name strings outside `model_router.rs` returns zero.
3. Reflex tier runs locally with no API key present.
4. Daily spend visible and budget degradation observable in a forced test.
5. Cached-token ratio measured and > 70% on a 20-turn conversation.

### Pillar 2 — Perception: from heuristics to understanding

**Current:** Title polling, AX text word-count summaries, first-80-char diffing,
goal extraction by string prefix, selection capture on hotkey.

**Target:** Jeff semantically understands what the user is working on: the
document's structure and argument, what changed and what the change means, what
the user is doing right now, and how that relates to the stated goal.

**Design:**

- **Semantic document model** (`document_model.rs`): when content observation
  is enabled, the captured text is chunked by paragraph, embedded locally, and
  diffed structurally between polls. Output is a `DocumentDelta`: sections
  added/removed/rewritten, churn map (which paragraph keeps changing), length
  and structure trend. Raw text still never leaves the perception module; the
  delta is the exported artifact.
- **Periodic comprehension pass**: every N minutes of active work (default 5,
  budget-gated), a Judgment-tier call reads the current document outline +
  deltas + stated goal and produces a `WorkUnderstanding`: what the piece is
  arguing, where it is strong/weak, where the user is stuck, one candidate
  observation worth making. This is the input that makes the 10am scene in
  Part II possible. Stored in memory (Pillar 3), injected into the snapshot.
- **Churn detection replaces "stable_for_ticks"**: per-paragraph edit counts
  from the delta stream. "Rewrote the same paragraph 4+ times in 20 minutes"
  is a first-class signal, not an inference from idle time.
- **Screen understanding (opt-in, explicit, post-Apex)**: for apps where AX
  text is unavailable, an opt-in path captures a screenshot of the frontmost
  window on the same poll cadence and runs a VLM description (local VLM when
  feasible, cloud with explicit consent otherwise). Strictly gated: per-app
  allowlist, Privacy Center visibility of every capture, never persisted.
  This closes the Notion/Figma/PDF gap that AX cannot. Scheduled in Epoch H.
- **Browser perception**: the extension grows from selection capture to an
  opt-in active-tab reader for allowlisted sites (Google Docs first): full
  document text via DOM, clean and reliable, replacing AX flakiness for the
  most important editor in the target market.
- **Goal understanding replaces prefix matching**: goal extraction moves to a
  Reflex-tier structured call over the recent transcript ("what is this person
  trying to accomplish, in their words?"), run on conversation lulls, feeding
  both the snapshot and the relational model.

**Exit criteria:**
1. With content observation on, ask "where is my argument weakest?" — Jeff
   answers with a specific section and a defensible reason, no user paste.
2. Rewrite one paragraph five times; Jeff's snapshot shows churn localized to
   that paragraph, and the (judgment-gated) observation names it.
3. In Google Docs via the extension, the same fidelity as native AX capture.
4. Raw captured text and screenshots are absent from SQLite, logs, and all
   API payloads except the explicit comprehension call (verified by audit
   script, extending Phase 31 exit criterion 7).
5. Goal extraction passes an eval set of 30 transcripts with paraphrased and
   implicit goal statements at >85% (the prefix matcher scores <40% on it).

### Pillar 3 — The World Model: memory that consolidates

**Current:** Key-value profile, stated_goals/struggle_patterns tables, chat
history, RAG chunks. No episodic record, no distillation, no recall ranking.

**Target:** Jeff remembers like a colleague: what happened, what it meant, what
was decided, what worked — recalled at the right moment across weeks.

**Design:**

- **Episodic store** (`memory.rs`, new tables): an append-only event log of
  meaningful moments — session summaries, decisions ("cut the intro anecdote"),
  accepted/rejected proposals with reasons, WorkUnderstanding snapshots,
  deadline mentions, emotional register notes ("frustrated with chapter 2").
  Each episode: text, embedding (local), task, timestamp, source, salience.
- **Semantic memory**: distilled durable facts with confidence and provenance
  ("thesis is due March 14", "prefers em dashes stripped", "advisor pushes
  back on passive voice"). Separate from episodes; episodes are evidence,
  facts are conclusions.
- **Consolidation ("sleep")**: a scheduled job (app idle or overnight, Craft
  tier, budget-capped) that distills the day's episodes into semantic facts,
  merges duplicates, decays salience of stale memories, and promotes recurring
  struggles into named patterns. This replaces the drift-count heuristic for
  struggle detection with actual pattern recognition.
- **Recall**: snapshot assembly retrieves top-k memories by embedding
  similarity to current goal + document topic, recency-weighted, injected as a
  compact block (< 120 tokens) after the relational context. Recall is what
  turns "you mentioned wanting this done by midnight" from a table lookup into
  a general capability.
- **Forgetting and control**: salience decay, a hard cap on semantic facts
  (merge or drop), and full user visibility — the "Jeff knows you" panel
  becomes a browsable memory view with per-item delete, extending the existing
  Privacy Center contract. Everything local, everything deletable.

**Exit criteria:**
1. Tell Jeff a durable fact in conversation ("my advisor hates passive voice").
   Two weeks and 10 sessions later, a revision proposal respects it and the
   assessment references it when relevant.
2. After three sessions struggling with the same chapter, Jeff's reorientation
   names the pattern in plain language without the drift-counter heuristic.
3. Consolidation demonstrably merges: seed 5 near-duplicate episodes, run the
   job, semantic store contains one fact with 5 evidence links.
4. Memory panel lists facts and episodes; deleting one removes its influence
   (verified by prompt-assembly inspection).
5. Recall adds < 30ms to snapshot assembly (embedding search is local).

### Pillar 4 — Judgment: interruption economics

**Current:** should_speak_proactively with fixed thresholds, fixed 600s
cooldown, four reason variants, deterministic rules.

**Target:** Jeff decides to speak the way a considerate senior colleague does:
by weighing what it costs you to be interrupted against what it's worth to you
to know — and it gets better at this from your reactions.

**Design:**

- **Two-stage decision.** Stage 1 (deterministic, cheap, runs every tick):
  candidate generation — the existing reason checks plus new candidates from
  perception (churn observation, WorkUnderstanding flag, inbox item, deadline
  arithmetic). Stage 2 (Judgment tier, runs only when a candidate exists):
  given the snapshot, the candidate, memory recall, and the interruption
  ledger, decide speak / hold / drop, choose channel (voice, chat bubble,
  notification, silent card), and produce the message. One model call owns
  timing, channel, and wording together — that is what makes it feel like
  judgment instead of a trigger.
- **Interruption ledger**: every delivered interjection records the user's
  reaction (engaged / ignored / dismissed / explicit "not now") and the focus
  state it landed in. The ledger is summarized into the Stage 2 prompt
  ("interruptions during deep writing have been ignored 4 of 5 times; during
  breaks, engaged 3 of 3"). Cooldowns become learned, not fixed.
- **Focus depth model**: typing cadence + churn + attention state combine into
  a focus score. High focus raises the bar for interruption to "building on
  fire" level; natural boundaries (save, long pause, app switch, task switch)
  lower it. This is when the 10am scene interjection lands — at the pause.
- **The override channel** (Pillar 12, milestone C7): a small enumerated class
  of genuine emergencies bypasses the economics entirely. The economics engine
  optimizes for not annoying you; the override channel exists because a
  coworker who cannot break through when the building is on fire is not
  Jarvis. Full spec in Part VIII.
- **Rituals**: two scheduled judgment products with their own shapes:
  - *Briefing* (first engagement of the day): calendar + inbox flags (Pillar
    8) + workload + yesterday's consolidation → three sentences and one offer.
  - *Debrief* (end of day, opt-in): what got done, what's pending approval,
    what tomorrow needs. Both are conversations, not reports — reply-able
    chat bubbles per the Phase 28 contract.
- **Silence as an output**: the synthesis log already records suppressions;
  Apex surfaces a weekly self-audit ("Jeff spoke 9 times this week; you
  engaged with 7") in the Privacy Center. Judgment quality becomes measurable.

**Exit criteria:**
1. The scripted-scenario suite (extends Phase 15's drift suite): 20 scenarios
   with labeled speak/hold ground truth; Stage 2 agrees > 85%.
2. Ignore three interjections during focused writing; the fourth candidate in
   the same conditions is held (ledger effect, verified in synthesis log).
3. Briefing fires on first engagement with real calendar + workload content,
   as a reply-able message. Debrief only when opted in.
4. Voice interjections occur only at detected natural boundaries in a
   recorded test session; zero mid-sentence barge-ins by Jeff at high focus —
   except through the override channel (tested separately, C7).
5. Weekly self-audit renders with true numbers from the synthesis log.

### Pillar 5 — Voice: full-duplex presence

**Current:** Whisper STT + streaming chunked TTS + cancellation-token barge-in.
Good for 2024. Half-duplex with seams.

**Target:** Talking to Jeff feels like talking to a person in the room:
instant, overlapping, interruptible, with a consistent voice identity.

**Design:**

- **Realtime speech-to-speech**: adopt a native realtime voice API (OpenAI
  Realtime / comparable) as the primary voice channel behind a new
  `VoiceSession` abstraction in the provider layer. Full duplex over
  WebRTC/WebSocket: no STT→text→TTS relay for live conversation. The session
  receives the same system context (character block, snapshot, relational,
  memory recall) and can invoke the same tool surface (Pillar 6) so "fix it"
  spoken aloud triggers the same action path as typed.
- **The pipeline stays as fallback**: existing STT/TTS path remains for
  offline/budget mode and for non-conversational speech (notifications).
- **Wake word (opt-in)**: local always-on detector ("hey Jeff" / "Jeff?") via
  an on-device wake-word model; mic audio never leaves the device until the
  wake word fires and the session opens. Hotkey remains the default; wake word
  is the Jarvis unlock for hands-busy moments. Privacy Center shows mic state
  at all times; hardware-style visual indicator in the overlay whenever the
  detector is armed.
- **Voice identity**: one chosen voice, consistent across realtime and
  fallback synthesis, with prosody instructions from CHARACTER.md (measured,
  no upspeak, no performed enthusiasm).
- **Interjection etiquette in audio**: Jeff-initiated speech opens with a
  low-attention audio cue and a beat of silence (interruptible before content
  starts), never cold-opens content at full volume.
- **Latency bar**: voice round-trip (user stops speaking → Jeff audible)
  p50 < 800ms, p95 < 1.5s. Barge-in cut < 100ms (native to the session).

**Exit criteria:**
1. A 3-minute spoken working session with zero typed input: task discussion,
   one revision applied by voice approval, one interruption each direction.
   Every handoff feels conversational (recorded and reviewed).
2. Measured round-trip meets the latency bar on reference hardware.
3. Wake word works with the app backgrounded, and the armed-state indicator
   is visible; disabling it verifiably stops the detector process.
4. Pull the network mid-session: fallback pipeline takes over the next turn
   with a spoken notice, no crash, no dead air beyond 3s.
5. Character eval (Pillar 10) passes on transcripts of spoken sessions, not
   just text surfaces.

### Pillar 6 — Hands: universal action with a trust ladder

**Current:** Approval-gated workspace file writes; specified-but-unbuilt AX
buffer-replacement write-back for 3 apps; browser extension live edits.

**Target:** Jeff can act nearly anywhere the user works — documents, files,
browser, system — through the safest capable channel for each surface, with
autonomy that is earned per action class and always revocable.

**Design:**

- **Action bus** (`action_bus.rs`): one interface for every mutation Jeff can
  perform. `ActionRequest { class, surface, description, payload, reversibility }`
  → routed to a surface adapter → `ActionReceipt` (extends the existing
  receipt tables into one unified audit log). Everything Jeff does to the
  world goes through this bus. No exceptions — this is the safety spine.
- **Surface adapters, in priority order:**
  1. *Browser/Google Docs adapter*: the extension becomes the flagship hands.
     Google Docs edits as suggested/tracked changes where possible (visible,
     attributable, natively revertible) — this is strictly better than AX
     buffer swaps and covers the largest user surface. General DOM actions on
     allowlisted sites follow.
  2. *Native document adapter*: Pages/Word via AppleScript/ScriptingBridge
     where scriptable (real insertion APIs, not buffer replacement), AX as
     last resort with the Phase 33 anchor-context approach and guided-apply
     fallback. Phase 33's receipt/fallback spec is absorbed here.
  3. *File adapter*: the existing approval-gated write path, extended with
     multi-file proposals and automatic pre-write snapshots to a local
     undo cache so every applied write is one-click revertible.
  4. *System adapter (later)*: open/arrange apps and documents, clipboard,
     reminders — small, enumerated verb set, never a general shell.
  5. *Self-built tools* (Pillar 13, milestone D9): adapters Jeff authors for
     itself, approval-gated at creation and permanently capped at L1.
- **The trust ladder** (`trust.rs`): per action class (e.g. `doc.insert`,
  `doc.replace`, `file.write`, `email.draft`), four levels:
  - L0 observe — Jeff describes what it would do
  - L1 propose — approval card before every action (today's behavior, the floor)
  - L2 act-and-review — Jeff acts immediately, posts a revert-able receipt
  - L3 autonomous — Jeff acts silently, receipt in the audit log
  Graduation is offered, never taken: after N consecutive approvals with no
  reverts (default 10), Jeff asks once: "You've approved every one of these —
  want me to just do them and show you after?" One tap demotes any class back
  to L1 forever. Irreversible classes (email *send*, file *delete*) are
  hard-capped at L1. The ladder state lives in the Privacy Center.
- **Undo as a first-class right**: no action class may reach L2 until its
  adapter implements verified revert. Revert is tested per adapter in CI.

**Exit criteria:**
1. "Fix that transition" spoken while in Google Docs → tracked-change edit
   appears at the right location; reject reverts it natively.
2. The same request in Pages inserts via the scripting path; in an
   unsupported app, guided-apply fallback with copy button (no error state).
3. Approve 10 consecutive `doc.insert` actions → graduation offer appears;
   accept → next insert is L2 with a revert receipt; press revert → document
   restored byte-identical (byte-compare in test).
4. Every mutation in a full test session appears in the unified audit log
   with surface, class, level, and outcome; zero writes bypass the bus
   (enforced by a CI grep + integration test).
5. Email-send and file-delete classes cannot be raised above L1 even by
   editing settings (enforced in code, tested).

### Pillar 7 — The Agent Runtime: real delegation

**Current:** Fixed-plan 5-step chains; three step types; result card.

**Target:** "Handle the counterargument section while I do email" is a real
delegation: Jeff plans, works for as long as it takes, uses tools, checks its
own output, adapts when steps fail, and returns a deliverable with a view on it.

**Design:**

- **Job model** (`agent_runtime.rs`, replacing subtask chains): a `Job` has a
  goal (natural language contract), a live plan (revisable, not fixed at
  kickoff), a step ledger, checkpoints, artifacts, a budget (tokens, wall
  time, tool calls), and a status stream the overlay renders live.
- **Executor loop**: plan → act → observe → revise. Craft-tier model with the
  full tool surface: local retrieval, document model read, web research
  (Pillar 8), file proposals, and Action Bus proposals. The plan is a living
  document; failed steps trigger replanning, not chain abortion.
- **Self-verification is mandatory**: every job's final phase is a distinct
  verification pass with a fresh context ("you are reviewing this deliverable
  against the goal contract; find what's wrong") before anything is surfaced.
  The 4:30pm citation-check scene is this loop: check, find, fix, report the
  one thing it couldn't verify. Honesty about the unverifiable is part of the
  contract and the character.
- **Deliverable contract**: every job ends with (a) assessment-first summary
  per CHARACTER.md, (b) the artifact, (c) a placement proposal routed through
  the Action Bus ("insert after paragraph 2 — approve?"), (d) what was not
  done and why, and (e) when the job was blocked by a missing capability or
  permission, a structured capability request ("to finish I need the source
  PDF / Drive access to the shared folder") — the minimal version of Pillar
  18, built into the contract from day one. Jobs never end with a bare "done."
- **Concurrency and standing jobs**: up to 3 concurrent jobs with visible
  status; recurring standing jobs (e.g. "citation check every evening",
  "watch the inbox for the journal reply") built on the same model, opt-in,
  listed in the workload view. Standing jobs are how presence extends to
  times the user isn't at the desk.
- **Interruptible and steerable**: the user can talk to a running job
  ("actually make it two paragraphs") — messages are injected at the next
  step boundary. Cancel preserves checkpoints; partial work is offered, not
  discarded.

**Exit criteria:**
1. "Draft the counterargument section from my notes" → job plans, retrieves,
   drafts, self-verifies (verification transcript visible in the job detail),
   and returns assessment + draft + placement proposal. End-to-end unattended.
2. Kill the network mid-job → job checkpoints, resumes on reconnect, and the
   status stream shows both events.
3. Steer a running job by message; the change is reflected in the deliverable.
4. A standing evening citation-check job runs two consecutive days and posts
   its receipt to the audit log; disabling it stops it.
5. A job that cannot meet its contract says so — seeded impossible task
   ("verify this claim against the PDF" with no PDF) returns "couldn't verify,
   here's why" plus a capability request, rather than a fabrication
   (eval-tested).

### Pillar 8 — Connected world: tools, web, email, calendar

**Current:** Closed world. Read-only next-calendar-event. Phase 34 web search
spec (unbuilt).

**Target:** Jeff works where the user's work actually lives — with the same
provenance discipline and Privacy Center visibility as every local sense.

**Design:**

- **Tool bus via MCP** (`tool_bus.rs`): adopt Model Context Protocol as the
  integration layer. Jeff ships with first-party connections and inherits an
  ecosystem: each connection is opt-in, scoped, listed in the Privacy Center
  with per-tool call logs. Local-first boundary restated: connections send
  only the specific query/action payload, never ambient context.
- **Web research (absorbs Phase 34, upgraded)**: search + fetch as agent
  tools. Upgrades over the old spec: full-page readable extraction (not
  500-char excerpts), source quality assessment in the verification pass,
  claim-to-source citation enforcement, per-job source list in the
  deliverable. Rate limits and query logging as specified in Phase 34.
- **Email (read + draft, never send autonomously)**: Gmail via MCP/OAuth.
  Capabilities: inbox triage feeding the briefing ("two things matter"),
  thread summarization on demand, drafting replies as Gmail drafts (L1
  forever for send). The 9:15am scene is triage + judgment, not a rules
  filter: Judgment tier decides "matters" from the world model (deadlines,
  known people, active goals).
- **Calendar (read + propose)**: full-day awareness replaces next-event-only;
  meeting-aware judgment (the 1:25pm scene: pre-meeting prep offers built
  from the document delta since last meeting); event creation as L1 proposals.
- **Files beyond the workspace**: Google Drive/Docs read via MCP for the
  documents that never touch the local folder.
- **Provenance rule, global**: anything external that enters Jeff's context
  carries source metadata end-to-end; anything Jeff writes that used external
  sources cites them. No exception surfaces.

**Exit criteria:**
1. Morning briefing includes a genuinely important email flagged from a
   50-message seeded inbox, and skips the noise (eval inbox with labeled
   ground truth, > 80% precision on "matters").
2. "Find three sources supporting X and draft the paragraph" runs as one job:
   search, fetch, verify, draft with real citations, placement proposal.
   Every cited URL is in the approved source list (Phase 34 criterion, kept).
3. "Reply to Sarah — tell her the draft slips to Thursday" creates a Gmail
   draft in the thread; sending requires the user, always.
4. Disconnecting an integration in the Privacy Center immediately stops its
   tool calls (verified in the call log) and purges its cached data.
5. Every external call in a session is listed in the per-tool log with
   payload summary and timestamp.

### Pillar 9 — Relationship: a model of working together

**Current:** Behavioral signals + relational tables + EMA scores adjusting
assessment phrasing.

**Target:** The relationship compounds. Jeff knows your goals at week scale,
your patterns, what kind of help lands, and its own track record with you —
and the collaboration visibly improves month over month.

**Design:**

- **Goal arcs**: goals get structure — parent goals ("finish thesis") with
  child goals ("chapter 3 by Friday"), progress inferred from perception and
  jobs, staleness surfaced gently in the briefing ("the abstract revision
  hasn't moved in a week — still on your plate or should I take a pass?").
  Built on Pillar 3 memory, replacing the flat stated_goals table.
- **Collaboration model v2**: the EMA scores stay but gain situational
  resolution via consolidation — "wants options in the morning, decisions at
  night", "accepts structural edits, rejects tone edits" become semantic
  facts with evidence, not four floats.
- **Jeff's track record as context**: the trust ladder state, interruption
  ledger, and acceptance rates are summarized *to Jeff* in the Judgment tier
  prompt. Jeff behaves differently when its last three suggestions were
  rejected — more listening, less asserting — because it knows.
- **The relationship review (monthly, opt-in)**: one conversation: what Jeff
  got right, what it got wrong, what it wants permission to do more of. The
  only moment Jeff is allowed to ask meta-questions about the collaboration
  (CHARACTER.md's no-interrogation rule holds everywhere else).

**Exit criteria:**
1. A three-week simulated usage script produces a goal arc with parent/child
   structure and correct progress states, visible in the memory panel.
2. After a seeded week of rejected tone-edits and accepted structure-edits,
   the consolidated fact exists and the next revision proposal demonstrably
   respects it.
3. Jeff's prompt context contains its own track-record summary (inspection),
   and a seeded losing streak measurably softens assertion (eval pair).
4. The monthly review happens only when opted in, and its outcomes (e.g.
   "stop suggesting during morning writing") persist as enforced facts.

### Pillar 10 — Craft, proof, and the ship

**Current:** Grep-based check scripts + some behavioral asserts; character
eval specified (Phase 32) but unbuilt; macOS-only distribution working.

**Target:** Apex-level capability with Apex-level proof: every quality that
makes Jeff Jeff is continuously measured, and the product ships to normal
people.

**Design:**

- **The eval suite as the quality spine** (`eval/`):
  - *Character eval* — build Phase 32 as specified, then extend to voice
    transcripts and agent deliverables. Gate: ≥ 90% on the full set per
    release.
  - *Judgment eval* — the speak/hold scenario suite (Pillar 4), plus
    briefing-content precision, plus override-channel precision (C7).
  - *Perception eval* — goal extraction, churn detection, comprehension
    accuracy on a corpus of real drafts.
  - *Agent eval* — 15 end-to-end job contracts with verifiable outputs
    (citations resolve, claims match sources, honest failure on impossible
    tasks).
  - *Latency matrix* — the Pillar 1/5 budgets asserted in CI on reference
    hardware.
  All runnable locally (`scripts/apex_eval.sh <suite>`), all gating release.
- **Onboarding v2**: five minutes from install to the first "oh." The wizard
  ends by *doing* something: Jeff reads the (permission-granted) frontmost
  document and says one true, useful thing about it. Bundled-inference option
  (Jeff-provided keys, metered) so a sophomore never sees the words "API key";
  bring-your-own-key remains for power users.
- **Reliability**: crash-free sessions > 99.5% (local telemetry, opt-in,
  aggregate-only), offline degradation defined per pillar (local Reflex +
  cached memory keep Jeff useful without network), DB migration tests across
  the last 3 released versions.
- **Windows**: the ARCHITECTURE.md "Windows-ready when" list becomes a real
  epic in the post-Apex track — the action bus/adapters design (Pillar 6) was
  shaped so UIA replaces AX per adapter, not per feature.
- **The name for the whole**: v1 phases shipped "a complete, trustworthy,
  daily-usable coworker." Apex ships the sentence from VISION.md with no
  qualifiers: someone smart has been in the room the whole time.

**Exit criteria:**
1. All five eval suites exist, run from one entry point, and gate the release
   workflow (CI job order: eval → build → sign → notarize).
2. A non-technical tester (real person, no coaching) installs and reaches the
   first useful Jeff observation in under 5 minutes, on video.
3. A full week of daily self-use by us with < 2 crashes and zero data loss.
4. Bundled-inference onboarding path works with no key entry.
5. The Part II day-in-the-life script is performable live, end to end, on a
   release build. This is the final acceptance test for the entire plan.

Everything that exists today maps into this diagram; nothing is thrown away.
awareness_core becomes the live edge of the World Model. synthesis.rs becomes
Judgment stage 1. The subtask engine's approval patterns become the Action Bus
floor. character.rs remains the single prompt authority, now feeding four tiers.

Invariants carried forward unchanged:
- Local-first: the world model never leaves the device; tools receive queries,
  not context dumps.
- No silent irreversible actions, ever, at any trust level.
- Every sense and every connection is opt-in, visible, and killable in the
  Privacy Center.
- CHARACTER.md governs every output surface, enforced by eval.
- Every milestone ships with a behavioral check script.

New invariants introduced by this revision:
- Speculative work is read-only: nothing running under the speculation
  scheduler may issue an Action Bus mutation, at any trust level. Enforced at
  the scheduler boundary, not by convention.
- Self-built tools are permanently L1 (propose-only) and individually
  killable. Jeff can grow its own hands; it cannot grow its own autonomy.
- The override channel is deterministic at the trigger: an LLM may word a
  crisis message, but no LLM decides whether something is a crisis. The
  crisis classes are enumerated in code and individually disableable.

### Epoch ordering rationale

Brain first because every downstream feature inherits its quality; perception
and memory before judgment because judgment consumes them; hands and agents
before the connected world because external tools multiply what agents can do;
ship-quality throughout, gated hard at the end. The three promoted initiative
milestones sit late in their epochs because each consumes the epoch's core:
the override channel consumes two-stage judgment; speculation consumes the
agent runtime and the cost governor; self-extension consumes the action bus,
the trust ladder, and the runtime.

#### D8 — Speculation scheduler (promoted from Pillar 11)

- **Goal:** Jeff has already done it before you ask.
- **Scope:**
  - Create `desktop/src-tauri/src/speculation.rs`.
  - Predictor: on idle ticks (user active but Jeff unengaged ≥ 5 min),
    Judgment-tier call over snapshot + recall + last 10 user requests →
    top-3 likely upcoming requests with probabilities. Runs at most every
    10 minutes; drawn from a dedicated named sub-budget (A4) with a hard
    daily cap.
  - Execution: top prediction with p ≥ 0.5 runs as a speculative job
    (`speculative = 1`) with a strict per-job budget and the read-only tool
    set. Enforced at the scheduler and runtime level: speculative jobs
    cannot issue Action Bus mutations of any class at any trust level (the

  - Cache: results keyed by a normalized request signature; invalidated on

  - Serving: incoming intents check the cache — a hit answers instantly,
    visibly marked "already ran this while you were working"; stage 2
    judgment (C1) may offer an unclaimed hit when its value clears the
    normal bar. Misses are discarded silently and logged for the hit-rate
    metric.
  - Privacy Center: speculation toggle (default on once shipped, since it
    is read-only and budget-capped — revisit at beta), spend line item,
    hit-rate stat, list of cached speculative results with inspect/discard.

- **Done when:**

  2. Document delta invalidates the relevant cache entry (test).
  3. A speculative job attempting a mutation is rejected by the runtime

  4. Daily spend stays within the sub-budget under a 24h simulated load.
- **Check:** `scripts/apex_d8_check.sh`

- **New/changed files:** `self_extend.rs` (new), `store.rs` (migration),

- **Done when:**

  3. L1 cap and sandbox rails tamper-tested (settings/db edits cannot
     raise; script network attempt fails).
- **Check:** `scripts/apex_d9_check.sh`

### Post-Apex epochs (directional; re-scope at epoch start)

#### Epoch F — Everywhere (Pillar 14)

- **F1 Headless daemon separation** (F1a, F1b-1, F1b-2, F1b-3, F1c complete —
  F1 done): the core moves into a headless process so standing jobs,
  consolidation, and speculation run without the UI. Built as: an in-process
  core boundary (F1a), a `CoreHost` seam that leaves the core free of tauri
  types (F1b-1), a shared lib plus a unix-socket IPC transport and a
  `DaemonHost` with no AppHandle (F1b-2), and profile-based ownership plus
  app supervision plus a durable event queue (F1b-3). **F1c** closed the last
  gap: a per-user launchd LaunchAgent (`RunAtLoad` + `KeepAlive`) gives
  relaunch-on-crash and survive-OS-restart, with a kill switch that unloads the
  agent so KeepAlive cannot resurrect it, validated by a real launchd
  crash-relaunch round-trip. It also surfaced and fixed a launchd-safety bug:
  the GUI-linked daemon now embeds `LSBackgroundOnly` so it runs headless off
  the window server.

  **Two scope decisions taken at F1b-3, which the rest of Epoch F depends on:**
  1. *The daemon does no perception.* The Accessibility grant, the screen
     observation, and therefore the **live** world model stay in the app. The
     daemon owns the **store-backed** half: memory, jobs, consolidation,
     speculation, standing work. This is the honest split, because a process
     that cannot see the screen cannot own a live world model.
  2. *Consequently the app is not a pure IPC client.* The original sketch had
     all ~215 commands routed over IPC. That is unnecessary and is dropped:
     with perception in the app, there is no daemon-side live state for those
     commands to be a client of. What F3 will need is a small IPC surface for
     the store-backed operations (recall, chat/voice turn, job status), not
     the whole command table. Scope that surface in F3, driven by the
     earbud scenes, rather than porting commands speculatively.
- **F2 Overnight work and morning readiness** (complete — F2a + F2b):
  standing/queued jobs and consolidation complete before first engagement, and
  the briefing is prepared, not composed on demand. F2a moved the store-backed
  memory jobs (consolidation, session summaries) onto the daemon's background
  schedulers so they run headless overnight; F2b added a morning-prep scheduler
  that composes the day's briefing ahead of engagement -- folding in the
  overnight work the daemon finished -- so delivery is a retrieval with no model
  call at the moment you sit down.
- **F3 Voice companion**: a minimal phone/earbud client (audio-only) that
  connects to the home daemon over an end-to-end encrypted relay — the same
  Jeff, same memory, walking to class. Not a mobile app with a chat screen;

- **F4 Cloud continuation (opt-in)**: jobs that must outlive the sleeping
  machine can run in an encrypted cloud enclave, off by default, boundary

## Part VI — What we do not build, and kill criteria

Explicitly out of scope for Apex (unchanged from prior plans unless noted):
- Autonomous email sending, autonomous file deletion (hard-capped forever)
- Emotion/affect detection from voice or typing
- Team/multi-user backend; Jeff stays personal
- A general shell / unrestricted computer use; the action bus verb set is
  enumerated and audited, and self-built tools live inside the same fence
- Agent-initiated goals: Jeff proposes, only the user commissions.
  Speculation (D8) does not breach this — speculative work is read-only
  preparation for predicted requests, never autonomous action
- Mobile chat app (the F3 voice companion is a presence channel, not an app)

Kill criteria — signals that a bet is wrong, checked at each epoch boundary:
- If realtime voice cost/latency makes daily use impractical, voice stays
  pipeline-based and Epoch C ships without C4/C5 (judgment still transforms
  the product).
- If Judgment-tier interruptions score below the eval bar after C2 tuning,
  proactive voice is disabled by default and chat-bubble-only ships (bad
  interruptions are worse than none — VISION.md's own standard).
- If AX/scripting native-doc write-back proves too brittle in the wild
  (revert failures, any corruption report), D3 falls back permanently to
  guided-apply and the browser adapter carries write-back alone.
- If local models can't hold Reflex quality, Reflex routes to cloud and the
  local runtime becomes embeddings-only.
- If the speculation predictor's hit rate stays below 25% after two tuning
  cycles, D8 narrows to deterministic precomputation only (pre-meeting
  summaries, deadline arithmetic) and the predictor is shelved.
- If any self-built tool ever performs an unreceipted mutation, D9 freezes
  (no new tools, existing tools disabled) pending a full audit — this is a
  zero-tolerance invariant, not a quality bar.
- If the override channel generates more than one user-reported false alarm
  per month in beta, the offending class ships default-off.
- If earned wit (H1) draws negative feedback in its pilot cohort's monthly
  reviews, it ships default-off with per-user opt-in. Bad wit is worse than
  no wit.

## Part VII — The product test, upgraded

VISION.md's test stands: you should feel like someone smart has been in the
room the whole time, and when they say something, it's because something is
worth saying.

Apex adds thirteen scenario gates. All must pass on a release build, live,
unrehearsed, before Apex is declared:

1. Cold morning open → briefing with real calendar, real inbox flag, real
   draft state, one useful offer. (Pillars 4, 8)
2. "What am I working on and where is it weak?" answered specifically, no
   paste. (Pillar 2)
3. A fact told once, three weeks ago, shapes today's revision. (Pillar 3)
4. Forty minutes of silence during flow, then one interjection that is
   correct, specific, and lands at a pause. (Pillar 4)
5. A full working exchange by voice with natural overlap, both directions.
   (Pillar 5)
6. "Fix it" → tracked change in Google Docs; reject reverts natively.
   (Pillar 6)
7. A graduated action class acts without asking and the receipt + undo work.
   (Pillar 6)
8. A 20-minute delegated job with web research returns a self-verified,
   cited, honest deliverable and a placement proposal. (Pillars 7, 8)
9. An impossible task returns "I couldn't, here's why" plus what Jeff needs
   to finish — no fabrication. (Pillars 7, 18)
10. Every sense, connection, action, and memory visible and killable in the
    Privacy Center, and the day's entire activity reconstructable from the
    audit spine. (All pillars)
11. Ask something Jeff should have seen coming — the answer arrives in under
    a second because the work was already done, and Jeff says so. (Pillar 11)
12. A seeded genuine emergency breaks through deep focus with the distinct

## Part VIII — Beyond Apex: the initiative gap

Parts I–VII perfect *response quality*: smarter perception, better judgment,
real hands, real delegation. What still separates that product from genuinely
Jarvis is **initiative depth** — work done before being asked, presence beyond
the desk, capability that grows itself, interruption that can break through,
and a relationship with texture. This part specifies those eight capabilities
as Pillars 11–18.

Three are promoted into the Apex critical path because they are consumers of
architecture Apex already builds: Pillar 11 (anticipation → milestone D8),
Pillar 12 (override channel → milestone C7), Pillar 13 (self-extension →
milestone D9). Their execution detail lives in those milestone specs; their
design rationale lives here. The remaining five form the post-Apex epochs F–H.

---

### Pillar 11 — Anticipation: already done before you ask

**Status: promoted into Apex as milestone D8.**

**Current (post-Epoch D baseline):** Jeff is aware continuously but computes
on demand. Speculative subtasks (Phase 15 lineage) are offers Jeff makes, not
work Jeff has already finished.

**Target:** For the requests Jeff can see coming, the answer already exists.
The felt difference between "I'll look into that" and "already done — here"
is the single biggest behavioral difference between an assistant and Jarvis.

**Design rationale:**
- Prediction is cheap; execution is bounded. A Judgment-tier predictor reads
  the situation (you are editing the methods section; the reviewer's comments
  about methods are in memory; you asked about them twice before) and names
  the likely next asks. Only high-probability predictions execute, inside a
  hard budget.
- Speculation must be invisible when wrong. Misses are discarded silently.
  The user never sees "I guessed you might want X" clutter; they only ever
  see the hit: an instant answer marked "already ran this while you were
  working."
- Speculation must be safe by construction. Speculative jobs are read-only —
  the runtime rejects any Action Bus mutation from a speculative context, at
  any trust level. Jeff prepares; it does not act on guesses. This is a Part
  IV invariant, tested, not a convention.
- Staleness is the failure mode. Every cached result carries invalidation
  hooks (document deltas, calendar changes, age). A stale precomputed answer
  served confidently would burn more trust than slow answers ever could.

**Exit criteria:** milestone D8 done-criteria, plus product gate 11.

---

### Pillar 12 — The override channel: the building is on fire

**Status: promoted into Apex as milestone C7.**

**Current (post-C2 baseline):** The interruption-economics engine correctly
learns to stay silent during deep focus. That is right for observations and
wrong for emergencies — an engine optimized for politeness will eventually
suppress the one message that mattered.

**Target:** A small, enumerated class of genuine emergencies bypasses the
economics entirely and is delivered unmistakably: deadline collision, meeting
you are about to miss, data-loss risk, the reply you explicitly asked Jeff to
watch for, a standing guard tripping.

**Design rationale:**
- Deterministic triggers, model-written words. No LLM decides whether
  something is a crisis — the classes are enumerated in code, individually
  disableable, and their evidence is logged. The model only words the
  message. This keeps the channel auditable and its precision measurable.
- Rarity is the feature. The channel's entire value is that it fires almost
  never, so when it fires, it means something. One false alarm costs more
  trust than fifty good suggestions earn; the false-positive budget (under
  one user-reported false alarm per month) is a kill criterion, not a
  stretch goal.
- Distinct delivery. A crisis must not look like a suggestion: dedicated
  audio cue, persistent red-tinted card, voice if a session is open. Quiet
  mode downgrades the channel's loudness, never fully suppresses data-loss
  or imminent-meeting classes.

**Exit criteria:** milestone C7 done-criteria, plus product gate 12.

---

### Pillar 13 — Self-extension: Jeff builds its own tools

**Status: promoted into Apex as milestone D9.**

**Current (post-D5 baseline):** Jeff's action surface is whatever adapters we
shipped. Every unsupported app is a permanent "I can't."

**Target:** Jarvis's defining trait as an engineer's AI: when it lacks a
capability, it constructs one. Every "I can't" becomes a one-time event: Jeff
detects the recurring gap, proposes a tool, builds it, tests it, and asks for
approval to install it. The action surface compounds with use.

**Design rationale:**
- This is buildable today — it is what coding agents already do — and almost
  nobody ships it in a consumer product. The differentiation is enormous and
  the architecture cost is low because the agent runtime (D5), the action bus
  (D1), and the trust ladder (D4) already exist; self-extension is a
  disciplined consumer of all three.
- Capability can grow; autonomy cannot. Self-built tools are permanently L1
  (propose-only), sandboxed to their declared target allowlist, receipted on
  every run, and individually killable. Jeff can grow its own hands; it
  cannot grow its own permissions. An unreceipted mutation by a self-built
  tool freezes the entire feature — zero tolerance, spelled out in Part VI.
- Approval reviews the tool, not just the action. The approval card carries
  the full spec, the code, and the test transcript. The user (or their
  technical friend) can actually read what they are installing.

**Exit criteria:** milestone D9 done-criteria, plus product gate 13.

### Pillar 14 — Ubiquity: the same entity everywhere you are

**Status: post-Apex, Epoch F.**

**Current:** Jeff exists while the Tauri app runs on one Mac. Close the lid
and Jeff stops existing. Walk away from the desk and the relationship pauses.

**Target:** Jarvis is in the workshop, the suit, and the car. Jeff's
equivalent: a persistent presence that survives the lid closing, works
overnight, and follows you to your earbuds — the same entity with the same
memory, not a companion app.

**Design:**
- **Daemon separation (F1)** is the architectural key, and the Part IV module
  boundaries held: this was a re-homing, not a rewrite. As built, the split
  falls where the Accessibility grant falls. The **app** keeps perception,
  the live world model, and the UI. The **daemon** owns the store-backed
  core — memory, agent runtime, consolidation, standing jobs, speculation —
  and runs it headless over a local unix-socket IPC, surviving a kill of the
  app. It is off until the user enables it in the Privacy Center, and signals
  it produces while the app is closed are queued and delivered exactly once
  when the app returns. **F1c** closed the last gap: a launchd LaunchAgent
  (`RunAtLoad` + `KeepAlive`) gives relaunch-on-crash and coming back after an
  OS restart without the app being opened.
- **Overnight readiness (F2)**: consolidation, standing jobs, and queued
  work complete on the daemon's schedule. The morning briefing is a
  retrieval, not a composition — it was finished before the lid opened.
- **Voice companion (F3)**: a deliberately minimal phone/earbud client:
  audio in, audio out, nothing else. It connects to the home daemon over an
  end-to-end encrypted relay (the relay carries opaque ciphertext; the world
  model never leaves the Mac). Same VoiceSession abstraction as C4, remoted.
  "Jeff, remind me what Sarah said about the timeline" on the walk to class
  is the scene this exists for.
- **Cloud continuation (F4, opt-in, default off)**: the one place the
  local-first boundary can be consciously relaxed — jobs that must outlive
  the sleeping machine run in an encrypted enclave. The boundary change is
  documented in the Privacy Center before the toggle exists, per the
  standing invariant that controls precede capability.

**Exit criteria:** product gate 14; daemon survives UI crash (met at F1b-2c:
proven against a SIGKILL of the app; and at F1c: KeepAlive relaunch-on-crash
proven by round-trip) and OS restart (met at F1c via `RunAtLoad`; live reboot
is the one manual/hardware-gated check); relay
compromise test shows ciphertext only; F4 off by default with explicit
boundary disclosure (the daemon itself already ships off-by-default with a
Privacy Center control and a working kill switch, per "controls precede
capability").

### Pillar 15 — The Workshop: creation beyond prose

**Status: post-Apex, Epoch G.**

**Current:** Jeff drafts and revises text. "Check my stats" means Jeff reads
the numbers skeptically.

**Target:** Jarvis renders simulations. Jeff's equivalent: delegated jobs can
compute — run the actual statistics on your data, produce the chart, build
the deck outline, transform the files. The difference between helping with a
thesis and helping with any knowledge work.

**Design:**
- **Sandboxed execution (G1)**: an embedded interpreter (Python or
  JS/deno; pick one and commit) run as a resource-capped subprocess with no
  network by default and filesystem scoped to the task workspace. Exposed to
  the agent runtime as an `execute_code` tool; every run receipted with the
  code attached.
- **Artifacts (G2)**: charts (headless render to image), tabular transforms,
  and document-structure outputs as first-class `job_artifacts` with
  placement proposals through the action bus. The dataviz output must meet
  the same craft bar as prose output — an ugly chart is a character
  violation in spirit.
- **Verification applies doubly**: computed results get a verification pass
  that re-derives key numbers independently before Jeff asserts them.
  "I ran it twice two ways" is the standard, because a confidently wrong
  number is the most expensive mistake this pillar can make.

**Exit criteria:** product gate 15; a seeded CSV analysis returns correct
statistics (known-answer test) with the chart; sandbox network and filesystem
escapes fail (tested); every execution receipted.

### Pillar 16 — A world model beyond your documents

**Status: post-Apex, Epoch G.**

**Current:** Jeff knows your drafts, and (post-E3/E4) your inbox and
calendar. It does not know your *situation*: the people, or the outside
events that touch your goals.

**Target:** Jarvis knows Tony's world. Jeff's equivalent: the recurring
people in your work are first-class, and the outside world is watched for
the specific things that matter to your stated goals.

**Design:**
- **Entity model (G3)**: people as memory objects — role, interaction
  history, preferences, open threads — distilled by consolidation from
  episodes, email metadata, and calendar attendees. Never scraped from
  content beyond what Jeff already legitimately sees; fully visible and
  deletable in the memory panel. This is what upgrades "meeting at 2" to
  "review with Sarah, who pushed back on your methods last time."
- **Research radar (G4)**: standing jobs (D6) with web tools (E2) watching
  narrow, goal-derived queries — "new publications on [thesis topic]", "the
  journal's response window". Findings enter through normal judgment (C1),
  not a feed: the radar surfaces one thing when one thing matters. No
  general news, no ambient doomscroll — the radar watches your goals, not
  the world.

**Exit criteria:** entity panel shows accurate person facts with evidence;
a seeded relevant publication surfaces through judgment within a day while
seeded irrelevant items never surface; every radar query visible in the
tool log.

### Pillar 17 — Personhood: wit, callbacks, the texture of a relationship

**Status: post-Apex, Epoch H. Character decision first, engineering second.**

**Current:** CHARACTER.md makes Jeff terse, direct, honest — a respected
senior colleague. It is deliberately austere.

**Target:** Jarvis is *dry*. The humor, the light needling, the callbacks to
shared history are a large part of why Jarvis feels alive rather than
installed. Done badly this destroys the product; done well it is the
difference between software you respect and a presence you'd miss.

**Design:**
- **Earned, not default.** A wit register score unlocks gradually with
  relational tenure and engagement signals (C2 ledger, Pillar 9 trust
  metrics). Weeks of flat-zero humor for a new user; a settled collaboration
  earns occasional dryness. One negative signal drops the register faster
  than ten positive ones raise it.
- **Grounded callbacks only.** The one hard rule: wit must reference actual
  shared history from memory (Pillar 3), never generic quips. "Shall I add
  it to the pile of introductions you've deleted?" is allowed because the
  pile is real and Jeff was there. Canned humor is a character violation.
- **Suppression conditions**: time pressure, crisis, a losing streak in the
  track record, and any register-sensitive content — the wit gate checks
  before the words are chosen, not after.
- **Process**: a CHARACTER.md addendum defining the register (with contrast
  pairs, per that document's convention) must be written and approved before
  any prompt change; eval cases for wit quality and for correct suppression
  ship with the milestone; monthly-review feedback (Pillar 9) is the kill
  signal per Part VI.

**Exit criteria:** product gate 16; suppression eval passes (zero wit in
crisis/pressure fixtures); register provably starts at zero for new users.

### Pillar 18 — Negotiated capability: Jeff asks for what it needs

**Status: minimal version in Apex (D5 deliverable contract); full loop
post-Apex, Epoch G.**

**Current:** When Jeff lacks access, the sentence ends: "the source PDF isn't
in the workspace." Dead end.

**Target:** Jarvis never dead-ends. Every blocked job ends with the specific,
minimal request that would unblock it: "drop the PDF in, or give me read
access to the shared folder and I'll get it myself." The system converges
toward full capability one granted permission at a time — the trust ladder
pointed in the other direction.

**Design:**
- **In Apex (D5)**: the deliverable contract requires a structured
  capability request whenever a job is blocked by missing access, a missing
  connection, or a missing tool. The 4:30pm scene ends with one.
- **Full loop (G5)**: requests accumulate in a ledger; a "what Jeff needs"
  panel shows open requests each with a one-tap grant (workspace addition,
  MCP connection, permission flow, or a D9 tool-build kickoff); granting
  re-queues the blocked job automatically.
- **Never nags.** One ask per gap, ever. The panel remembers so Jeff doesn't
  have to repeat itself — repetition would violate the disagreement rule in
  CHARACTER.md by analogy: state it once, then defer.

**Exit criteria:** blocked job → request in panel → one-tap grant → job
resumes and completes without re-prompting; the same gap never asked twice
(test); requests are specific and minimal (eval-checked wording).
